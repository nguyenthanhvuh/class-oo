  1. Consider the "int enum pattern", a throwback to C, vs. Java enums. 
    // int enum pattern (in Java)
public static final int APPLE_FUJI         = 0;
public static final int APPLE_PIPPIN       = 1;

public static final int ORANGE_NAVEL       = 0;
public static final int ORANGE_TEMPLE      = 1;

vs.

// Java enum - basic version
public enum Apple  { FUJI, PIPPIN };
public enum Orange { NAVEL, TEMPLE };
    Bloch identifies various problems with the first pattern. Identify 3 of these reasons.







    Answer
    • The int enum pattern is not type safe. This means that comparing apples to oranges: 
      (APPLE_FUJI == ORANGE_TEMPLE)
      is meaningless, but compiles and runs. On the other hand, the expression: 
      (FUJI == TEMPLE)
      won't compile.
    • The name spaces for enums are distinct, hence you are stuck with APPLE_FUJI in the int enum pattern, instead of the more convenient FUJI.
    • The int enum pattern is brittle. Since integer constants are compiled in, changing a constant requires recompiling clients, which is basically unrealistic in real-world programs. On the other hand, Java enums are objects, so dynamic dispatching yields the correct behavior without client recompilation.
    • There is no easy way to obtain Strings from the int enum pattern, but Java enums provide a sensible default behavior in toString().
    • There is no way to determine, at run time, how many enums are in an int enum pattern, but this information is easily available in Java enums.
    Grading note: Variations of these answers are acceptable for grading, as are answers that reference the added expressive power of Java enums. Only the reason is required; explanations are not.
